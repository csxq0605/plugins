# Team-Coord Integration Guide for dep-audit

## 并行审计策略

dep-audit 的多阶段可以并行执行，由 team-coord 的 lead 协调。

## Worker 分配

### 分配方案 A: 按审计类型

```
Lead:
  "我需要你审计这个项目的依赖安全。3 个 worker 分别负责：
   Worker A → 查询所有依赖的已知漏洞（OSV API）
   Worker B → 检查过期依赖
   Worker C → 检查许可证风险"
```

### 分配方案 B: 按生态系统

适用于多语言项目：

```
Lead:
  "我需要你审计这个 monorepo 的依赖。每个 worker 负责一个生态系统：
   Worker A → npm 依赖（package.json）
   Worker B → PyPI 依赖（requirements.txt）
   Worker C → crates.io 依赖（Cargo.toml）"
```

## 合并策略

Lead 收到所有 worker 的结果后：

1. **合并漏洞列表** — 按严重程度排序，去重
2. **合并过期列表** — 按版本差距排序
3. **合并许可证风险** — 按风险等级分组
4. **计算综合风险评分** — 加权求和
5. **生成修复优先级** — 先 Critical 漏洞，再 High，依此类推

## 风险评分公式

```
风险评分 = 100 - (Critical×25 + High×10 + Medium×3 + Low×1)
```

## 输出模板

Lead 合并后使用以下模板：

```markdown
# Dependency Audit Report

{worker_results_merged}

## 综合修复计划

| 优先级 | 包名 | 问题 | 修复方式 | 命令 |
|--------|------|------|----------|------|
| P0 | {pkg} | Critical CVE | 升级 | `{cmd}` |
| P1 | {pkg} | High CVE | 升级 | `{cmd}` |
| P2 | {pkg} | 过期 3 个大版本 | 升级 | `{cmd}` |

---
*Generated by dep-audit with team-coord*
```

## 注意事项

- Lead 不做实际查询，只做分配和合并
- Worker 之间不需要通信，各自独立查询
- OSV API 无速率限制，但建议控制并发
- 如果某个 worker 失败，Lead 用自己的 context 补充
